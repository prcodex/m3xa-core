# LESSONS

Append-only log of **structural** mistakes the project has burned hours on. The *why* behind the conventions BODY.md describes.

Skip ordinary bugs. Capture lessons whose root cause was a structural choice you wouldn't have known to look for. Add a new entry when a fix surfaces a pattern, not when you fix a typo.

## 2026-05-19 · Mermaid clicks need inline code blocks, not SVG `<img>` embeds

**Lesson:** GitHub-rendered Mermaid diagrams only honor `click NodeId "url"` directives when the diagram is in an inline ```` ```mermaid ```` block. SVG embeds rendered via `<img src="diagram.svg">` lose the interactivity.

**Why it matters:** Spent an afternoon "adding click handlers to the SVG" before discovering GitHub's markdown renderer simply doesn't run JS on embedded images. The fix was trivial — replace the `<img>` with the inline Mermaid source — but the wrong mental model cost hours.

**How to apply:** When you want a clickable diagram on GitHub, embed the Mermaid source directly. Don't pre-render to SVG.

## 2026-05-19 · Vendor SDKs stay inside `backends/`

**Lesson:** Never `import anthropic` (or `voyageai`, or `lancedb`) from an actor, scraper, or self-awareness component. They go through the `LLMBackend`/`EmbeddingBackend`/`VectorDBBackend` protocols in `m3xa_core/backends/`.

**Why it matters:** The whole point of "pluggable backends" collapses the first time a non-backend module imports the vendor SDK directly. Once that happens you're stuck with that vendor until you grep-and-replace every call site. Caught early it's a one-line discipline; caught late it's a refactor.

**How to apply:** When adding a new feature that calls an LLM or embedder, route it through the protocol. If the protocol doesn't expose what you need, extend the protocol — don't reach around it.

## 2026-05-19 · Router output is a schema contract, not a suggestion

**Lesson:** The router returns `{"expertises": [...], "confidence": 0.0-1.0, "rationale": "..."}`. Every downstream actor reads this shape. Treat it as a published API.

**Why it matters:** Once you change the router output, you change every assembler/agent_hub/synthesizer that consumes it. The contract is what lets the actors stay independently testable.

**How to apply:** Adding a new field to the router output? Update `m3xa_core/schemas.py` first, then every consumer that reads it, then write a test. Removing one is even worse — grep for the field name across the repo before deleting.

## 2026-05-19 · Per-tool pointer files all point at the same source of truth

**Lesson:** `CLAUDE.md`, `.cursor/rules/m3xa.mdc`, `.github/copilot-instructions.md`, `AGENTS.md` should all be **thin pointers** to BODY.md + concepts/. Don't repeat content across them.

**Why it matters:** Duplication rots. The day BODY.md's "read this first" section drifts from CLAUDE.md's, both files lose credibility. Single source of truth + N doorways is the pattern.

**How to apply:** When a new coding tool gains popularity, add a thin pointer file in whatever location it expects. Five lines max. Funnel the reader to BODY.md.

## 2026-05-19 · BODY.md autogen with timestamp churn requires content-diff check

**Lesson:** The `meta` section of BODY.md contains a regeneration timestamp + commit SHA. Including the timestamp in the diff would make every regenerator run "out of sync" with itself. Fix: regenerate non-meta sections first, only update meta if non-meta content actually changed.

**Why it matters:** First version of the regenerator updated the timestamp every run, and the `--check` CI mode then failed every PR. Took 20 minutes to figure out the loop.

**How to apply:** Any autogenerated content that includes timestamps or commit SHAs needs a "did anything else change?" check before the timestamp is updated.

## 2026-05-20 · Anonymization is a CI check, not a pre-publish review

**Lesson:** When the goal is a public repo with no real source names, enforce it at every commit via a hashed blocklist in CI — not via a human review before publishing.

**Why it matters:** A single PR six months from now that mentions a real bank name slips past a tired reviewer. CI doesn't get tired. The check is `tools/check_anonymization.py`; the blocklist is SHA256-hashed at `.anonymization-blocklist.sha256` so readers can't reverse it.

**How to apply:** Generate the blocklist locally from `.anonymization-blocklist.txt` (which is gitignored), hash each line, commit the hash file. CI reads the hash file at PR time and rejects any matching token in the diff.

## 2026-05-21 · Regex anonymization checks substring-match in non-obvious ways

**Lesson:** A blocklist of bank/expert/wire names without word boundaries will substring-match inside ordinary English words. `ubs` matches inside `pubs`, `Substack`, `subset`. Five false positives blocked a real publication and required a manual decision.

**Why it matters:** The check fails *loudly* on false positives, which costs trust in the gate. Worse, the temptation to ignore future flags grows. The fix is two-layered: tighten the regex with `\b` word boundaries, AND add an AI second-pass for high-stakes gates that catches what the literal-string check misses (indirect references, novel mentions).

**How to apply:** Routine pre-commit → `tools/check_anonymization.py` (hash-based, fast, deterministic). Pre-publication gate → also run `tools/check_anonymization_ai.py` (one Claude call, ~$0.02, catches indirect identifications a blocklist can't enumerate). Both layers, belt + suspenders, for any public push from a private system.
